## 1. Objective

Two backfill goals in a single sequential pass:

1. **Article backfill** — Ingest articles from 14 new sources for each day in the
   date range into `public.newsapi_articles`.
2. **Event backfill** — Collect events for **all** articles (existing 14-source
   corpus + newly ingested 14-source articles) by switching the event collector
   to `time_window` discovery scope during the backfill.

After the backfill, update the production `EVENTREGISTRY_SOURCE_URIS` to include
all 28 sources going forward.

## 3. Architecture: How the Backfill Works

### Pipeline Daisy Chain

Each day's backfill triggers the full four-stage pipeline:

```
aws lambda invoke (article collector)
       │
       ▼  writes manifest.json to S3
  ┌──────────────┐
  │ article_loader│ ◄── S3 trigger on manifest.json
  └──────┬───────┘
         │  writes load_report.json to S3
         ▼
  ┌──────────────────┐
  │ event_collector   │ ◄── S3 trigger on load_report.json
  └──────┬───────────┘
         │  writes event-manifest.json to S3
         ▼
  ┌──────────────────┐
  │ event_loader      │ ◄── S3 trigger on event-manifest.json
  └──────┬───────────┘
         │  writes event-load-report.json to S3
         ▼
     CYCLE COMPLETE ── script advances to next date
```

### Key Env Var Overrides Per Lambda

**`newsapi-ai_collector`** (article collector):

| Variable | Backfill Value | Purpose |
|---|---|---|
| `BACKFILL_LOCAL_DATE` | `YYYY-MM-DD` (iterates daily) | Fetch exactly this Honolulu calendar day |
| `RUN_TYPE` | `backfill` | Tags pipeline_runs and articles as backfill |
| `EVENTREGISTRY_SOURCE_URIS` | 14 new sources (comma-separated) | Only fetch new sources |
| `COLLECTOR_MODE` | `date_window` | Normal date-based collection (unchanged) |

**`event_collector`** (event collector):

| Variable | Backfill Value | Purpose |
|---|---|---|
| `EVENT_DISCOVERY_SCOPE` | `time_window` | Scan ALL articles in window, not just parent run |
| `EVENT_DISCOVERY_TIME_COLUMN` | `date_time_published` | Match by publication date, not ingestion time |
| `EVENT_DISCOVERY_START` | Day start in UTC | Computed from Honolulu date |
| `EVENT_DISCOVERY_END` | Day end in UTC | Computed from Honolulu date |
| `EVENT_DISCOVERY_ONLY_MISSING` | `true` | Only fetch events not already in DB |
| `RUN_TYPE` | `backfill` | Tags run as backfill |

**`newsapi-ai_loader`** and **`event-loader`**: No env var changes needed — they
are triggered by S3 and operate on whatever artifacts they receive.

### Why `time_window` Solves Both Goals

The `time_window` discovery query (from `event_collector_lambda.ts` line 317-326):

```sql
SELECT DISTINCT a.event_uri
FROM public.newsapi_articles a
WHERE a.date_time_published >= $1
  AND a.date_time_published < $2
  AND a.event_uri IS NOT NULL
  AND a.event_uri <> ''
LIMIT $3
```

This has **no filter on `run_id`, `ingestion_source`, or `run_type`** — it scans
all articles in the time window. Combined with `EVENT_DISCOVERY_ONLY_MISSING=true`,
it will:

- Pick up event_uris from existing 14-source articles that were never collected
- Pick up event_uris from newly loaded 14-source articles
- Skip events that already exist in `public.events`

## 4. Script Strategy

### 4.1 Pre-flight

1. Snapshot current Lambda configurations for all 4 functions
2. Validate AWS CLI access and permissions
3. Determine the S3 artifact path pattern for polling

### 4.2 Main Loop (28 iterations: 2026-02-14 → 2026-03-13)

For each date:

1. **Compute time boundaries**
   - Honolulu (Pacific/Honolulu, UTC-10, no DST) date → UTC window
   - Example: `2026-02-14` → `2026-02-14T10:00:00Z` to `2026-02-15T10:00:00Z`

2. **Update article collector env vars**
   - Merge `BACKFILL_LOCAL_DATE`, `RUN_TYPE=backfill`,
     `EVENTREGISTRY_SOURCE_URIS=<14 new sources>` into existing env vars
   - `aws lambda update-function-configuration --function-name newsapi-ai_collector ...`
   - `aws lambda wait function-updated --function-name newsapi-ai_collector`

3. **Update event collector env vars**
   - Merge `EVENT_DISCOVERY_SCOPE=time_window`,
     `EVENT_DISCOVERY_TIME_COLUMN=date_time_published`,
     `EVENT_DISCOVERY_START=<utc_start>`, `EVENT_DISCOVERY_END=<utc_end>`,
     `EVENT_DISCOVERY_ONLY_MISSING=true`, `RUN_TYPE=backfill`
   - `aws lambda update-function-configuration --function-name event_collector ...`
   - `aws lambda wait function-updated --function-name event_collector`

4. **Invoke article collector**
   - `aws lambda invoke --function-name newsapi-ai_collector --payload '{}' /tmp/response.json`
   - Check response for `FunctionError`

5. **Poll for pipeline completion**
   - Poll S3 for `event-load-report.json` (or `load_report.json` if no events
     were discovered) at the expected artifact path
   - Polling interval: 30 seconds
   - Timeout: 15 minutes (safety net)

6. **Display results summary for the completed date**
   - Print article counts (fetched, deduped, inserted, updated) from load_report
   - Print event counts (discovered, fetched, inserted, updated) from event-load-report
   - Print any warnings or errors from CloudWatch logs
   - Query `pipeline_runs` and `pipeline_run_metrics` for the backfill run

7. **Operator review gate — PAUSE and wait for approval**
   - The script **stops and waits for operator input** before proceeding
   - Operator reviews the results and chooses one of:
     - **`y` / `yes`** — Approve and continue to the next date
     - **`r` / `retry`** — Re-run the same date (e.g., if partial failure)
     - **`s` / `skip`** — Skip this date and move to the next
     - **`q` / `quit`** — Abort the backfill, restore Lambda configs, and exit
   - This gate ensures no date proceeds without human validation
   - The script will **not auto-advance** — it blocks until input is received

8. **Log approval decision and advance to next date**

### 4.3 Post-backfill

1. **Update `newsapi-ai_collector` source list** to all 28 sources
2. **Restore `event_collector`** to `parent_run` scope (production default)
3. **Clear `BACKFILL_LOCAL_DATE`** and reset `RUN_TYPE=scheduled`
4. Wait for function-updated on both Lambdas
5. Print summary report

## 5. Timeout Analysis

**Article collector** (300s timeout):
- ~2,468 articles/day across 14 sources at 100/page = ~25 pages
- ~3-5 seconds per page (API call + processing) = ~75-125 seconds
- Heavy days (news.yahoo.com spikes) could push to ~150-200 seconds
- **Verdict: 300s should be sufficient** for per-day collection

**Event collector** (300s timeout):
- Discovers event_uris from all articles on a given day (both 14 old + 14 new sources)
- Fetches events in batches of 50
- With `ONLY_MISSING=true`, volume decreases as backfill progresses
- **Verdict: Should be fine**, but monitor for days with unusually high event counts

**Risk mitigation**: If any single day fails or looks wrong, the operator can
choose to **retry** that date or **quit** the backfill entirely at the review
gate. The idempotent upsert design means retries are safe.

---

## 6. Idempotency & Safety

- **Articles**: Upserted by `uri` primary key. Re-running a date produces the
  same result. `IS DISTINCT FROM` prevents unnecessary updates.
- **Events**: Upserted by `uri` primary key. `first_collected_at` is preserved,
  `last_collected_at` updates.
- **Pipeline runs**: Each invocation gets a unique `(run_id, ingestion_source,
  run_type, nth_run)` tuple. Re-runs increment `nth_run`.
- **Env var restoration**: Script snapshots configs before starting and restores
  on exit (including on Ctrl+C via trap).

## 7. Rollback

If the backfill needs to be reversed:

```sql
-- Remove backfilled articles
DELETE FROM public.newsapi_articles
WHERE run_type = 'backfill'
  AND date_time_published >= '2026-02-14T10:00:00Z'
  AND date_time_published < '2026-03-14T10:00:00Z'
  AND source_uri IN ('axios.com','bloomberg.com','abcnews.com','cbsnews.com',
    'us.cnn.com','latimes.com','nbcnews.com','newsweek.com','theguardian.com',
    'thehill.com','usatoday.com','news.yahoo.com','rt.com','jpost.com');

-- Events are shared across sources, so generally should not be deleted.
-- Remove backfill pipeline_runs records if needed:
DELETE FROM public.pipeline_runs
WHERE run_type = 'backfill'
  AND run_id >= '2026-02-14T10:00:00Z'
  AND run_id < '2026-03-14T10:00:00Z';
```

## 8. Monitoring

During the backfill, monitor:

- **CloudWatch Logs** for each Lambda function (errors, timeouts)
- **S3 artifact paths** for manifest.json / load_report.json presence
- **Script stdout** — the script logs each date's start, completion status,
  and article/event counts from the response payload
- **`pipeline_runs` table** — query for `run_type = 'backfill'` to see status
- **`pipeline_run_metrics` table** — per-stage metrics for each run

```sql
-- Monitor backfill progress
SELECT run_id, status, articles_fetched, articles_deduped, rows_loaded,
       db_rows_inserted, db_rows_updated
FROM public.pipeline_runs
WHERE run_type = 'backfill'
ORDER BY run_id;
```
